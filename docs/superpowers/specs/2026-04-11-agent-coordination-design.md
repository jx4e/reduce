# Agent Coordination Skill Design

**Date:** 2026-04-11  
**Status:** Approved

## Overview

A personal skill (installable globally at `~/.claude/skills/`) that gives parallel Claude agents a shared coordination protocol to prevent file-level merge conflicts. The skill has two sides: an orchestrator protocol for declaring file ownership, and an agent protocol for checking claims before touching files.

## Problem

When multiple Claude agents work in parallel on the same codebase, they can unknowingly edit the same files, producing merge conflicts or overwritten work. The existing `dispatching-parallel-agents` skill advises keeping agents scoped, but provides no mechanism for agents to actively verify they're not conflicting.

## Design

### Registry Location

`~/.claude/agent-registry/<project-slug>-<session-id>.json`

- `project-slug`: sanitized absolute project path (slashes → hyphens), e.g. `Users-jake-Projects-reduce`
- `session-id`: timestamp + short random suffix, e.g. `20260411-a3f2`, generated by the orchestrator at dispatch time

Each orchestration run gets its own session file. Old files accumulate harmlessly and can be deleted at will.

### Registry Format

```json
{
  "session": "20260411-a3f2",
  "project": "/Users/jake/Projects/reduce",
  "agents": {
    "agent-auth": ["app/api/auth/**", "lib/auth.ts"],
    "agent-ui": ["components/**", "app/(main)/**"]
  }
}
```

Patterns are glob-style strings. Agents reason about matches by inspection (prefix match for `dir/**`, exact match for specific files). No code execution required — Claude can determine overlap from the pattern strings directly.

### Orchestrator Protocol

1. Generate session ID: `date +%Y%m%d-$(openssl rand -hex 2)` (e.g. `20260411-a3f2`)
2. Ensure `~/.claude/agent-registry/` exists: `mkdir -p ~/.claude/agent-registry`
3. Decide file ownership for each agent based on task decomposition
4. Write registry to `~/.claude/agent-registry/<slug>-<session>.json`
4. Pass to each agent in their prompt:
   - The registry file path
   - Their agent name (key in the registry)
   - Their claimed paths (redundantly, for clarity)

### Agent Protocol

On startup:
1. Read the registry file
2. Confirm own claimed paths match the registry entry for this agent name
3. Before touching any file during the task: verify no other agent's patterns match it
4. If conflict detected: **hard stop** — do not touch the file, report back with:
   - The conflicting file path
   - Which agent owns it per the registry
5. If registry file is missing or malformed: proceed with a warning in the response

### Conflict Response Format

When stopping on a conflict, agents return:

```
COORDINATION CONFLICT: Cannot proceed.

File: components/auth/LoginForm.tsx
Claimed by: agent-ui (pattern: components/**)
This agent: agent-auth

Stopped before making any changes to the conflicting file.
Completed work on non-conflicting files:
- lib/auth.ts (updated token validation)

Orchestrator action needed: re-assign or serialize this file.
```

### What the Skill Does NOT Do

- No file locking (Claude can't do atomic locks)
- No real-time agent-to-agent communication
- No auto-resolution of conflicts
- No enforcement — agents follow the protocol voluntarily

### Skill Structure

```
~/.claude/skills/coordinating-parallel-agents/
  SKILL.md    # Single file, self-contained
```

The skill is split into two clearly labeled sections agents can navigate to:
- `## Orchestrator: Setting Up the Registry`
- `## Agent: Checking the Registry`

## Out of Scope

- Dynamic claim expansion mid-task (agents stop if they need an unplanned file)
- Time-based expiry of claims
- Cross-machine coordination
